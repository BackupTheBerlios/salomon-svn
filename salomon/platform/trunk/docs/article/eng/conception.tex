\section{Conception of the \emph{Salomon} platform}

The proposed architecture of the \emph{Salomon} platform ia a replay
to some limitation of \emph{VINLEN}.
The process of gathering the knowledge from data is usually very complex
-- some algorithms require large amount of computation 
which are being taken on the considerable amount of given data.
From here efficiency problems are appearing and therefore 
while designing the architecture of the system 
its scalability was also one of more important aspects.
We managed to attain it thanks to breaking the entire computational process 
for isolated tasks and it created the possibility of dispersing calculations.

Main assumptions of the platform are:

\begin{description}
    \item[Component-based design.]
    \emph{VINLEN} is a complete product, delivered together with different algorithms built in.
    \emph{Salomon} is an open platform which is supposed to make a base for other projects 
    using the Machine Learning idea in the process of discovering the knowledge in data. 
    Hence the assumption, that platform -- main module of the system -- 
    is as simple as possible. Providing the user-friendly environment is setting it for 
    the realization of the functionality contained in components having the form of so-called plug-ins.

    \item[Open platform.] The user has the full freedom in adding new components (plug-ins) to the system.

    \item[Independence from the runtime environment.]
		Thanks to using portable tools, for example Java, the system can be started on any system having 
		the appropriate virtual machine.
    \item[Possibility of calculations distribution.]
		Thanks to component-based design and right definition of tasks, 
		he system is enabling the work in the distributed, heterogeneous network environment.
    \item[Simplicity of using.]
    An easiness of using is one of bases of the \emph{VINLEN} system.
    \emph{Salomon} tries also in this respect to equal and what is more: to outrun his progenitor.
    One of more important facilitation being expected is replacing the language of defining ,,scouts'' 
    with graphical interface in the form of the graph of the the knowledge flow.
\end{description}

System \emph{Salomon} consists of two components. 
First from them is a platform. The second part is a set of plug-ins, 
providing logic needed for configuring, executing and showing the results of execution.
The platform consists of two elements: the engine and the data storage.
The first of them is responsible for managing and executing tasks 
and the second is serving for the data and knowledge storage.
A possibility of elastic defining the course of the processing 
is one of essential advantages of the system. 
It is realized in the form of so-called \emph{tasks}.

An extraction of the knowledge is a process which can be divided in stages 
-- in every stage the knowledge is being subjected to sure processing 
(e.g. attributes are discretized, the rules are created, the knowledge is tested). 
Since every stage is creating the separate integrity 
and the part of stages is independent of oneself, so they can be processed simultaneously.

A task is a basic individual representing calculations.

It is defined as:
$$
z=(a, p, in, out)
$$
gdzie:
\begin{itemize}
    \item $a$ is an algorithm, provided to the system in the form of the component -- pluing,
    \item $p$ represents parameters of the algorithm,
    \item $in$ and $out$ are the input and output information of the algorithm.
\end{itemize}

Input and output can have the form of data (in the case of algorithms which accomplishes selection of data),
of knowledge (e.g. generated rules) or of data and the knowledge 
all at once (e.g. rules and exceptions for them).

We expect that the exit of one task can be an access of certain number of next tasks.
So that the task can be made, all tasks on which it is depending must be made earlier.
Connected objectives can be introduced in the form of the directed graph.
Such a graph with the emphasized initial task is creating the \emph{program}.
Additionally, every edge can have the assigned condition which is required 
so that steering is handed over in length to this edge. 
This way the defined program can easily be introduced the user 
in the graphical form. 
In the system it will be possible also to define agents 
which will be reacting to changes in data and/or of the knowledge
(e.g. to adding the new record to training data 
or generating the knowledge by the different agent).
