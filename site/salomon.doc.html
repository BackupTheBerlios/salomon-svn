System agentowy

Opis systemu

Zadaniem 'systemu' jest agregowanie pluginów, wywolywanie je w odpowiedniej kolejnosci, przechowywanie danych, zbiorów regul oraz umozliwianie pluginom operowanie na nich. Funkcjonalnosc jest przeniesiona do pluginow, zadaniem systemu jest tylko zarzadzanie ich wykonaniem.

Przed wywolaniem akcji plugin otrzymuje instancje klasy Environment, w której zapisany jest aktualny stan systemu. Przez stan ten rozumiemy aktualnie wybrany Rule Set lub Data Set i wszystkie inne informacje ktore moga byc wykorzystywane przez pluginy. Stan ten moze modyfikowac uzytkownik lub plugin. Na stan ten sklada sie np. aktualny zbiór regul, aktualne dane oraz obiekt klasy Engine, dzieki której moze operowac na tych danych. Kazdy plugin moze zmodyfikowac Environment tak, ze nastepny plugin bedzie dzialal na zmodyfikowanej wersji srodowiska.

Knowledge system

Klasa odpowiada za: zarzadzanie baza danych i zbiorami regul zarzadzanie odczytem i zapisem danych w bazie udostepnia potrzebna pluginom funkcjonalnosc poprze Engine

IGraphicalPlugin

Interfejs ktory implementuja pluginy graficzne. Pozwala pobrac parametry od uzytkownika, które nastepnie zostana przekazane do pluginow ? zwraca je im za pomoca metody getSettings().

IDataPlugin

Posiada jedna metoda doJob() przyjmujaca jako parametry Environment, czyli aktualny stan systemu oraz obiekt Settings reprezentujacy ustawienia pluginu.

ISettings

Interfejs reprezentujacy ustawienia dla pluginu.

Environment

Klasa przechowywujaca wszelkie dane zwiazane z systemem. Ma forme slownikowa ? dane przechowywane sa w formie obiektów dostepnych pod odpowiednimi kluczami. Pozwala na dostep do danych, regul a takze np. zbiór aktualnych pluginow, dane uzytkowika itp.

Engine

Klasa udostepniajaca wszelka potrzebna funkcjonalnosc systemu pluginom. Wraz z Environmentem pozwala pluginom na wykonanie dowolnej czynnosci w systemie. (?)

Description

Klasa pozwalajaca zdefiniowac specyfikacje pluginu. Moze to umozliwiac sprawdzenie poprawnosci sekwencji pluginów poprzez porównanie

Przykladowa realizacja pluginu

Sam sytem jest w stanie pobrac parametry od uzytkownika i uruchomic pojedynczy plugin. Najwazniejszym pluginem jest Scheduler. Jego zadaniem jest zarzadzanie uruchamianiem innych pluginów. Bedzie on potrafil tworzyc kolejki, w których zapisuje kolejnosc wywolania pluginow i klucze do ich ustawien (na poczatku wywolywane sa formatki pozwalajace zdefiniowac ustawienia dla wszystkich pluginów, które beda pózniej wywolywane). Nastepnie uruchomi je w odpowiednij kolejnosci przekazujac im jako parametry pobrane wczesniej od uzytkownika ustawienia.

Zastosowane technologie

System budowany jest z wykorzystaniem serwera baz danych Firebird.
Implementacja zostanie wykonana w jezyku Java JDK1.5 z wykorzystaniem RMI.
